## 1. 기존 프로젝트 소스코드의 문제점

업무 화면 개발·로컬 구동·성능 점검 과정에서 확인된 문제를 영역별로 정리하였습니다.

### 1.1 전년도예산조서적용 화면 (기능·UI)

- **조회 방식이 과도하게 무거움**
  - 6단계 트리 전개, FRSC(재원), CHILD_CNT 등 매핑 목록에 불필요한 연산 포함
  - 세세목만 필요한 업무인데 전체 트리 구조를 매번 생성하여 응답 지연

- **매핑·적용 업무 흐름 부재**
  - 전년도 1건 ↔ 올해 1건을 묶어 관리하는 매핑 목록 UI 없음
  - 건별 적용만 가능하여 다수 세세목 적용 시 반복 작업 필요

- **화면과 맞지 않는 조건·코드 잔존**
  - JSP에서 제거한 통계목·재원구분·지시일자 필터를 JS에서 계속 로드·검증
  - 트리 그리드용 formatter·변수 등 미사용 코드 다수 (~1,350줄)

- **조회 버튼 검증 오류**
  - 전년도 조회 시 적용대상(올해)의 `condReportCd`를 검사하는 버그 존재

- **금액 표시와 업무 요구 불일치**
  - 목록에 전년도예산액(PRE_BGT_AMT) 중심 표시 → 실무상 **조정액(DIFF_AMT)** 확인 필요

### 1.2 백엔드·SQL

- **경량 조회 API 부재**
  - 기존 `ajaxBudgetCopyNewList.do`만 존재 → 무거운 `selectCopyReportList` 호출
  - 매핑용 평면 목록 전용 쿼리·엔드포인트 없음

- **일괄적용 API 부재**
  - 단건 `copyReport`만 호출 가능, 여러 매핑을 한 번에 처리하는 API 없음

- **SQL 비효율**
  - `TB_DGRTEMNGMOK` 통계목명을 동일 서브쿼리로 2회 조회
  - UI에 쓰지 않는 OFFICE_NM, DEPT_NM, PRE_BGT_AMT 등 불필요 컬럼 SELECT
  - 재원 EXISTS 조건 등 화면에서 사용하지 않는 필터 SQL 유지

- **코드 중복**
  - Controller 목록 API 간 사용자·권한 처리 로직 반복
  - Service/DAO 양쪽에서 `reportTableNm` 중복 설정
  - `copyReportBatch`에서 `copyReport` 로직 재작성

### 1.3 공통 필터 (AccessLogFilter) — 전역 장애

- **증상:** 로그인·조회 등 **모든 AJAX**에서 "조회에 실패하였습니다" 발생

- **원인 1 — TB_COMM_SEQ 미시드**
  - loaddb 적재 후 `TB_COMM_SEQ`가 비어 tracelog ID 채번 실패

- **원인 2 — 필터 처리 순서**
  - tracelog INSERT 실패 시 예외로 빠져 `JSON_PARAM` request attribute 미설정
  - 요청 본문(InputStream)은 이미 소비된 상태 → Controller가 빈 JSON 파싱

### 1.4 DB·운영 스크립트

- **인덱스 부족**
  - loaddb·DDL만 적용 시 조회용 인덱스 미적용 → 대용량 JOIN·필터 지연

- **인덱스 적용 스크립트 한계**
  - `csql -i create-indexes.sql` 실행 시 **첫 중복 INDEX에서 전체 중단**
  - CUBRID는 `CREATE INDEX IF NOT EXISTS` 미지원

- **loaddb 후 후처리 누락**
  - 데이터 적재 스크립트에 인덱스·채번 시드 자동 실행 없음

- **로컬 구동 환경**
  - 메뉴·사용자·채번 데이터 부재 시 화면·로그인 불가 (별도 시드 필요)

### 2.1 전년도예산조서적용[신규] — 조회·화면

| No | 기존 사항 | 개선 사항 |
|----|-----------|-----------|
| 1 | 6단계 트리 전개 쿼리(`selectCopyReportList`)로 목록 조회 | 세세목(COMPO_LEVEL='1')만 조회하는 **경량 평면 쿼리** `selectCopyNewMapList` 추가 |
| 2 | 트리 그리드(jqGrid tree) | **평면 csGrid** — 제목 + 조정액 2컬럼 중심 |
| 3 | 전년도예산액(PRE_BGT_AMT) 표시 | **조정액(ADJ_AMT = DIFF_AMT)** 표시로 업무 요구 반영 |
| 4 | 좌·우 패널 각각 무거운 트리 조회 | `/budget/ajaxBudgetCopyNewMapList.do` **경량 API**로 양쪽 패널 조회 |
| 5 | 통계목·재원구분·지시일자 조건 UI·JS 유지 | JSP·JS에서 **해당 필터·콤보·검증 제거**, 필요 조건만 유지 |
| 6 | 트리용 formatter·미사용 변수 ~1,350줄 JS | **~911줄**로 정리, `buildMapSearchData` / `validateMapSearchCond` 공통화 |
| 7 | 전년도 조회 시 올해 reportCd 검증 버그 | 검증 로직 통합·**올바른 condPrefix** 기준으로 수정 |
| 8 | 매핑 목록·일괄적용 UI 없음 | 하단 **매핑 그리드**, 매핑추가/삭제/전체삭제, **일괄적용** 버튼 추가 |
| 9 | 동일 올해 대상 중복 매핑 가능 | **teBgtCompoId 기준 중복 추가 차단** |
| 10 | 매핑 영역 좁음 | south 패널 **240px(160~400px 리사이즈)**, 목록·매핑 영역 높이 자동 조정 |

### 2.2 백엔드·API

| No | 기존 사항 | 개선 사항 |
|----|-----------|-----------|
| 1 | 목록 API 1종(무거운 트리) | **경량** `ajaxBudgetCopyNewMapList.do` 추가 (기존 API는 호환 유지) |

| 6 | DAO·Service 모두 reportTableNm 설정 | **Service에서만** `reportTableNm` 설정, DAO 중복 제거 |
| 7 | 미사용 import·주석 dead code | BudgetCopyService, BcjisCommVO 등 **정리** |

### 2.3 SQL (BudgetCopyNew.xml)

| No | 기존 사항 | 개선 사항 |
|----|-----------|-----------|
| 1 | TB_DGRTEMNGMOK 서브쿼리 2회 (제목·DGRCOMPO_NM) | **OUTER JOIN 1회** `(+)` 로 통계목명 조회 |
| 2 | OFFICE_NM, DEPT_NM, COMP_GROUND, PRE_BGT_AMT 등 다수 SELECT | 그리드·적용에 필요한 **최소 컬럼만** SELECT (응답 크기 감소) |
| 3 | FRSC EXISTS (재원구분 필터) | UI 미사용 → **조건 제거** |
| 4 | TB_DGRCOMPO 선행 후 REPORT 조인 | **REPORT 테이블 선필터** 후 COMPO_LEVEL='1' 조인 (인덱스 활용 유리) |
| 5 | 매핑용 전용 쿼리 없음 | `BudgetCopyNew.selectCopyNewMapList` **신규 정의** |

## 4. 기대 효과

- **업무 효율:** 전년도→올해 매핑 목록 작성 후 **일괄적용**으로 반복 작업 감소
- **응답 속도:** 경량 쿼리 + 인덱스로 **목록 조회 시간 단축**
- **안정성:** AccessLogFilter·TB_COMM_SEQ 보완으로 **AJAX 전역 장애 해소**
- **유지보수:** JS·Controller 중복 제거, 운영 스크립트 자동화로 **배포·재구축 부담 감소**
